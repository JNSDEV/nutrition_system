<decisions>
## Implementation Decisions

### `library/meals.md` restructure (BLOCKER-01)
- **D-01:** **H2 = archetype, H3 = meal name.** `## Breakfast`, `## Lunch`, `## Dinner`, `## Snack` are H2; each meal under them is an H3 (e.g. `### Protein Overnight Oats`). The H3 heading text's kebab-case form IS the anchor (`library:meals#protein-overnight-oats`).
- **D-02:** **Per-meal `archetype:` inline field, immediately under the H3.** Format: `**archetype:** breakfast | lunch | dinner | snack`. Used by `/swap-meal` Step 5 archetype filter. Same value as the H2 it sits under (redundant but explicit so command parsing doesn't depend on heading hierarchy).
- **D-03:** **Per-meal estimated macros are optional in v1.** If present under a meal as a one-line `**est:** kcal=NNN, p=NN, c=NN, f=NN`, commands will use them as kcal_estimate inputs. If absent, commands fall back to "estimate by ingredient lookup" (already specified in command bodies). Do not block the restructure on missing numbers.
- **D-04:** **No semantic content rewrite.** Migrate existing bullet-list meals into H3 sections; do not invent new meals or remove existing ones. The job is structural.

### `library/recipes.md` restructure (BLOCKER-01)
- **D-05:** **H2 per recipe category, H3 per recipe.** Same pattern as meals: `## Breakfast Recipes`, `## Dinner Recipes`, etc. (categories may differ from meals if the existing file groups differently — preserve existing groupings as H2). Each recipe is an H3; kebab-case of the H3 = `library:recipes#{anchor}`.
- **D-06:** **Per-recipe ingredient block format.** Under each H3, an `**ingredients:**` line followed by a bulleted list, one ingredient per line. Format: `- {qty} {unit} {ingredient}` (e.g. `- 2 cups rolled oats`, `- 30 g whey isolate`). `/shopping-list` D-18 aggregation expects this shape; planner picks how strict to be on units (`g`, `cups`, `tbsp` are all fine; just be consistent within a recipe).
- **D-07:** **4-portion baseline preserved.** Existing recipes are sized for 4 portions (Phase 1 convention). Do NOT rescale during the restructure.

### `library/macro-templates.md` mapping (BLOCKER-02)
- **D-08:** **Add a `## Session-Type Mapping` table at the top of the file** (before the existing template bodies). Columns: `calendar Session label | session_type token | archetype token | macro template heading`. Concrete mappings (audit recommendations — planner verifies against actual calendar contents):

  | Calendar Session label | session_type | archetype | Template |
  |----|----|----|----|
  | `OFF` / `Rest` | `rest` | `lower_carb` / `protein_forward` | Rest Day Template |
  | `Z1 commute` | `easy_ride` | `lower_carb` | Rest Day Template (or trim version) |
  | `Z2 Zwift indoor` | `medium_ride` | `moderate_carb` | 60 min Z2 Template |
  | `Long Z2` / `long ride` | `long_ride` | `high_carb` | Long Z2 Template |
  | `SST` / `Tempo` / `Intensity` | `intensity` | `high_carb` | SST / Tempo Template |
  | Race day | `race` | `high_carb` | Long Z2 Template (or race-day variant) |

  Planner reads the actual `calendar/cycling-2026.md` Session column values and finalizes the mapping; the table above is a starting point.
- **D-09:** **CAL-02 contract update required.** `library/cal-02-contract.md` lookup chain: cycling row → `session_type` (NEW field, derived via D-08 table) → archetype → macro-template heading. The contract MD file may need a one-line edit to point at the new mapping table. Planner decides whether to edit the contract or just rely on the mapping table being self-explanatory.
