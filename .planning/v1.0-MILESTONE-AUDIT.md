#### BLOCKER-01 — `library/meals.md` has no H2/H3 headings; library-anchor contract cannot be executed

**Requirement affected:** CMD-01, CMD-02, CMD-03, CMD-04, CMD-05, CMD-06 (all commands)
**Cross-phase contract:** Phase 2 D-04 / D-12 → Phase 3 all commands

**What the contract requires (Phase 2 D-04, Phase 3 D-04, `docs/conventions.md` Section 6):**

> "Anchor derivation: take the H2 or H3 heading in `library/meals.md`, convert to kebab-case."
> "Resolution procedure: Read `library/meals.md`. Find the H2 or H3 heading whose kebab-case form equals the anchor."

Every command file that references meals emits or consumes `library:meals#{anchor}` format and
instructs the model to resolve anchors by matching H2/H3 headings in `library/meals.md`.

**What `library/meals.md` actually contains:**

The file is a flat bulleted list under a single `# Meals` H1. There are no H2 or H3 headings
for individual meals. The full body is:

```
Breakfast:
- Protein overnight oats
- Warm protein oats
...
Dinner:
- Chicken + rice + vegetables
...
```

The colon-suffixed category labels (`Breakfast:`, `Lunch:`, `Dinner:`, `Snacks / dessert:`) are
plain text lines, not markdown headings.

**Impact:** The anchor-resolution procedure defined in Phase 2 and encoded in all 6 Phase 3
commands cannot execute. There are no headings to match against. Every `library:meals#{anchor}`
reference emitted by `/weekly-plan`, `/log-day`, `/shopping-list`, `/swap-meal`, and
`/prep-today` is unresolvable by the documented procedure.

The `/swap-meal` filter (Step 5: "same archetype as the meal slot", "filter meals.md by archetype")
also cannot execute because there are no per-meal sections with archetype metadata.

**Broken flows:** Sunday weekly-plan, Daily log, Mid-day swap, Monday morning prep, Monday weekly-review.

**Fix required:** Restructure `library/meals.md` to use H2/H3 headings per meal, with kebab-case
anchor-compatible names, and optionally add per-meal archetype and macro estimate fields.
This is a library content change, not a command change.

#### BLOCKER-02 — `library/macro-templates.md` has no `session_type` labels; commands cannot map calendar rows to macro archetypes

**Requirement affected:** CAL-02, CMD-01, CMD-02, CMD-03, CMD-05, CMD-06
**Cross-phase contract:** Phase 2 CAL-02 contract (`library/cal-02-contract.md`) → Phase 3 commands

**What the contract requires:**

From `library/cal-02-contract.md` Sources of truth:
> "`library/macro-templates.md` — authoritative for macro archetypes per session_type. Phase 3 reads at runtime and matches archetype by `session_type`."

Every command that resolves CAL-02 (all except `/shopping-list`) instructs:
> "Read `library/macro-templates.md` — macro archetype matching today's `session_type`."

The `calendar/cycling-2026.md` standard-week table provides session labels like:
`Z1 commute`, `Intensity (VO2 / Threshold, alt.)`, `Z2 Zwift indoor`, `OFF`, `Fasted Z2 + Z1 commute`, `Strength (full body)`, `Long Z2 outdoor`.

**What `library/macro-templates.md` actually contains:**

The file has headings `Rest Day Template`, `60 min Z2 Template`, `SST / Tempo Template`,
`Long Z2 Template` for Jonas, and `Standard Cut Day` / `Lighter Day` for Partner/Farva.

There is no `session_type` field in the file. There is no mapping table from calendar session
labels to archetype labels. The command instructions also reference archetypes `high_carb`,
`lower_carb`, `carb-heavy`, and `protein-forward` (per `weekly-plan.md` Step 4 Rule 3), but
none of these archetype tokens appear in `library/macro-templates.md`.

**Impact:** The matching step "find macro archetype by `session_type`" has no defined resolution
path. A model executing the commands would need to infer a mapping that is not documented
anywhere. The CAL-02 contract says macro-templates.md is authoritative for this mapping, but the
file doesn't contain it.

**Broken flows:** All flows that pass through CAL-02 target resolution (all except `/shopping-list`).

**Fix required:** Add a `session_type` → archetype mapping table to `library/macro-templates.md`,
or document the mapping explicitly in `library/cal-02-contract.md`.

#### WARNING-01 — `library/calorie-targets.md` uses "Partner" not "Farva"; CAL-02 Farva-branch formula is ambiguous

**Requirement affected:** CAL-02, CMD-02, CMD-05, CMD-06
**Cross-phase link:** Phase 2 D-01 (Farva name resolution) → Phase 3 commands → Phase 4 terminology sweep

The D-09 terminology sweep (Phase 4 plan 04-05) was scoped to `.planning/` files only (PROJECT.md,
ROADMAP.md, REQUIREMENTS.md). It did not touch the operational library files.

`library/calorie-targets.md` line 17 reads `Partner:` — this is the label commands must read to
derive Farva's static kcal target. Commands instruct "read `library/calorie-targets.md` (Farva
branch)" but the file has no "Farva" section — only a "Partner" section.

Same issue in:
- `library/macro-templates.md` line 37: `Partner` header for Farva's macro templates
- `library/portions.md` line 18: `Partner:` for Farva's portion sizes
- `library/recipes.md` line 18: `Partner:` for Farva's recipe quantities
- `library/README.md` lines 18, 21, 41: "Partner" in active descriptions

A model running `/log-day`, `/swap-meal`, or `/weekly-review` for Farva following the command
instructions ("read calorie-targets.md Farva branch") will fail to find a "Farva" section. It
will either infer the "Partner" section is Farva's, or report that the section is missing.

**Not a hard block** (the model will likely infer the correct section) but the mismatch between
command prose ("Farva branch") and file content ("Partner") is a reliability risk on every
Farva-scoped command execution.

#### WARNING-02 — `cycling-2026.md` has no `session_type` column; resolution procedure is implicit

**Requirement affected:** CAL-01, CAL-02, CMD-01, CMD-02, CMD-05
**Cross-phase link:** Phase 1 calendar → Phase 2 CAL-02 contract → Phase 3 commands

`calendar/cycling-2026.md` has columns `Day`, `Session`, `Duration`, `Est. kcal`. There is no
`session_type` column.

The CAL-02 contract states: "session_type: string lifted from cycling-2026.md". The `Session`
column contains labels like `Z1 commute (both legs)`, `Intensity (VO2 / Threshold, alt.)`,
`Z2 Zwift indoor`, `OFF`. These are the implied `session_type` values.

Commands instruct the model to "find today's row for Jonas: `session_type` and `training_est_kcal`"
but the column is called `Session`, not `session_type`. The `cal-02-contract.md` bridges this
("session_type: string lifted from cycling-2026.md") but the bridge depends entirely on model
inference. There is no explicit mapping or column header alias documented.

For Sunday, two tables exist (standard week: "Long Z2 outdoor (variable)" + Sunday progression by
date range). The contract documents the resolution rule ("Day-of-week → standard-week table;
Sunday → Sunday-progression table by date range, per Phase 1 D-08") but Phase 1 D-08 is a
planning decision, not a file. The calendar file itself has no instruction about which table
governs which day.

**Risk:** Marginally fragile — the model must synthesize across two tables without explicit
instruction in the calendar file itself.

## Tech Debt / Carried Forward

1. **BLOCKER-01 (HIGH):** `library/meals.md` and `library/recipes.md` must be restructured to use H2/H3 headings per meal/recipe before any command can execute the documented anchor-resolution procedure. This is the most impactful single gap in the system.

2. **BLOCKER-02 (HIGH):** `library/macro-templates.md` needs an explicit `session_type` → macro archetype mapping. Options: (a) add a mapping table to the file, (b) add a `session_type` alias column to `calendar/cycling-2026.md`, (c) add the mapping to `library/cal-02-contract.md`. Any one of the three would close the gap.

3. **WARNING-01 / WARNING-05 (MEDIUM):** "Partner" → "Farva" rename across the 8 operational library files that the Phase 4 terminology sweep missed. Priority files: `library/calorie-targets.md`, `library/macro-templates.md`, `library/portions.md`, `library/recipes.md`.

4. **WARNING-02 (LOW):** Consider adding a `session_type:` column or a resolution note to `calendar/cycling-2026.md` so the standard-week → session_type → macro-archetype chain is explicit rather than inferred.

5. **WARNING-03 (LOW):** Correct `docs/conventions.md` Section 1 — remove `/swap-meal` from the "Read by" column for `trackers/weekly-plans/YYYY-Www.md`.

6. **WARNING-04 (LOW):** Update `REQUIREMENTS.md` and `ROADMAP.md` checkboxes to reflect delivered state. 13 requirement IDs and both ROADMAP phase rows need `[x]` toggles.

## Recommended Action

Before archiving v1.0, resolve BLOCKER-01 and BLOCKER-02. These are library content gaps, not
command logic gaps — the commands are correctly specified. The fix is:

1. Rewrite `library/meals.md` with H2/H3 headings per meal (e.g. `## Chicken + Rice + Vegetables`)
   and add per-meal archetype metadata (breakfast/lunch/dinner/snack) and rough macro estimates.
2. Rewrite `library/recipes.md` with H2/H3 headings per recipe (e.g. `## Protein Overnight Oats`).
3. Add a session_type mapping section to `library/macro-templates.md` linking calendar labels to
   macro archetypes.
4. Sweep "Partner" → "Farva" in the four library files and `library/README.md`.

Items 3–6 in Tech Debt are cleanup and can follow, but the system cannot execute a single complete
E2E flow until BLOCKER-01 and BLOCKER-02 are resolved.
